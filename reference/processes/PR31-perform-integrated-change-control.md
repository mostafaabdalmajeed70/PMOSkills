# PR31 — Perform Integrated Change Control

## Process Identity
| Field | Value |
|---|---|
| **Process ID** | PR31 |
| **Process Name** | Perform Integrated Change Control |
| **Process Group** | Monitoring and Controlling |
| **Knowledge Area** | Integration Management |
| **PMBOK Source** | Process Groups: A Practice Guide §7.2; PMBOK 8 §2.1.6.8 |

---

## Purpose
Review all change requests; approve changes and manage changes to deliverables, project documents, and the project management plan; and communicate the decisions. Allows documented changes to be considered in an integrated manner while addressing overall project risk.

---

## Inputs
| Input | Description |
|---|---|
| Project management plan | Change management plan, configuration management plan, scope/schedule/cost baselines |
| Project documents | Basis of estimates, change log, requirements traceability matrix, risk report |
| Work performance reports | Performance data from Monitor and Control Project Work |
| Change requests | Formal proposals to modify any document, deliverable, or baseline |
| Enterprise environmental factors | Regulatory requirements, organizational governance |
| Organizational process assets | Change control procedures, templates |

---

## Tools and Techniques
| Tool / Technique | Notes |
|---|---|
| Expert judgment | Evaluating impact of proposed changes |
| Change control tools | Configuration management system, automated workflows |
| Data analysis | Alternatives analysis, cost-benefit analysis |
| Decision making | Voting, autocratic decision making, multicriteria decision analysis |
| Meetings | Change Control Board (CCB) sessions |

---

## Outputs
| Output | Description |
|---|---|
| Approved change requests | Changes approved for implementation via Direct and Manage Project Work |
| Project management plan updates | Any component updated as a result of approved changes |
| Project documents updates | Change log updated with disposition of all requests |

---

## Key Concepts
- Performed throughout the project from start through completion.
- Before baselines are established, changes do not require formal control.
- Once baselined, all changes go through this process.
- Change requests may be approved, deferred, or rejected.
- A **Change Control Board (CCB)** may be used for larger or more complex projects.
- Applies to both **predictive** (formal change requests) and **adaptive** approaches (backlog management for adaptive).

---

## PMBOK 8 Alignment
- Corresponds to **Assess and Implement Changes** process in PMBOK 8 §2.1.6.8.
- Integrated change control is a core mechanism for maintaining the **Governance** performance domain.
- Supports the principle: *Navigate Complexity* and *Enable Change*.

---

## Common Pitfalls
- Allowing verbal changes without formal documentation.
- Bypassing the CCB for convenience.
- Failing to assess cross-domain impact (scope, schedule, cost, risk, resources).
